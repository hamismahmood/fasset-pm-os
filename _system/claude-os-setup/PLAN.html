Decisions to Make Before Starting
Decision 1 — Project folder migration strategy

The OS pattern uses projects/<slug>/. iClaude uses numbered folders (01_Global_USD_Account, 02_Statement_of_Account, etc.).

Recommended: Keep numbered folders exactly as-is. Create a projects/ directory with symlinks to each numbered folder using clean slugs (keel-iban → ../01_Global_USD_Account). Scripts detect the OS from 00_agents/ + projects/ presence. No renames, no data moves. Migrate folders to native format opportunistically when you start new projects.

Decision 2 — Graphiti: now or Phase 2?

Graphiti is what makes session hooks powerful — cross-session memory with semantic search. Without it, the hooks fall back to MEMORY.md (which already works fine).

Recommended: Skip Graphiti in Phase 1. The hooks are written to gracefully fall back to MEMORY.md. Add Graphiti as Phase 2 once the base OS is running smoothly. It requires Docker + Neo4j + OpenAI key — non-trivial setup.

Decision 3 — CLAUDE.md split

Currently all context (universal preferences, PM scope, project rules, voice) lives in iClaude/CLAUDE.md. The OS model splits this: universal stuff goes in ~/.claude/CLAUDE.md, OS-specific stuff stays in iClaude/CLAUDE.md.

Recommended: Split it. Universal rules (verify before claiming, one question at a time, no em dashes, PKT timezone, no filler) go to ~/.claude/CLAUDE.md. PM scope context, file routing table, project index, scheduled task rules — stay in iClaude/CLAUDE.md. The OS CLAUDE.md gets shorter and cleaner.

Decision 4 — Permission mode

The OS install sets "defaultMode": "bypassPermissions". Your current setting is "auto".

Recommended: Keep "auto" for now. Bypass mode skips all confirmation prompts — powerful but risky for destructive operations. You can always switch it later once you're comfortable with the agent roster.

Decision 5 — PM agent roster

The template agents are engineering/consulting focused (agent-architect, system-analyst, etc.). Your context is PM at a crypto fintech.

Recommended: Adapt 2 templates (devils-advocate, discovery-agent — both are actually PM-useful as-is). Write 3 net-new agents: pm-strategist (prioritisation, product decisions), spec-writer (PRDs, Jira tickets, requirements), stakeholder-comms (Slack, email, vendor comms in your voice). Install all 5 universal templates too — they're useful in any OS.

Risks
MEDIUM
Session hooks slow down session start
Hooks have a 10s timeout. The session-start hook is lightweight (no network calls without Graphiti). Low actual risk, but monitor it.
MEDIUM
CLAUDE.md split creates duplication
If we're not careful, universal rules end up in both files. Plan: only move rules that genuinely apply across any working directory (not just iClaude). Clear ownership criteria needed.
LOW
Scheduled tasks break after .claude/ layer is added
The Scheduled/ folder runs as a Cowork routine with its own context. Adding .claude/ to iClaude shouldn't affect it — but confirm in Phase 4 testing.
LOW
install-os-claude.sh install_path detection uses ~/Documents/
The session-start hook assumes OS folders live under ~/Documents/. iClaude is on ~/Desktop/. The hook will not auto-detect the OS group ID. Minor issue — can patch the hook to also check ~/Desktop/.
Recommended Order
Suggested sequence

Run Phases 1-3 in a single session (roughly 2 hours total). Phase 4 testing takes 30 minutes and should be done in a fresh Claude Code restart. Phase 5 (Graphiti) is genuinely optional and can wait weeks — the OS works fine without it.

Start with Phase 1 step 1.6 last — the CLAUDE.md split is the highest judgment call and benefits from seeing the hooks/agents working first so you know what context Claude actually has access to at session start.
